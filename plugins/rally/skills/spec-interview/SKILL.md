# spec-interview:仕様の引き出し

人間は仕様を事前に書き下せない。具体物を見て初めて「これじゃない」が分かる。
だからこのスキルでは、AI が観点表に沿って質問し、人間は回答と反例出しだけを行う。

## 絶対のルール

1. **ユーザーが実装開始を指示するまで実装しない。** 壁打ち(質問と要件整理の往復)が
   終わってプランがまとまっても、実装に入るのは「実装して」等の明示指示があったときだけ。
2. **質問は 1 ラウンドあたり最大 5 問。** 観点表の全項目を聞くのではなく、「答えによって
   実装が最も変わるもの」を選別する。足りなければ次のラウンドで聞く。
3. **仕様は抽象語ではなく具体例で確定する。** 「新しい順に並べる」ではなく「田中(3/1 登録)と
   佐藤(3/5 登録)なら佐藤が上」の形まで落とす。

## 手順

壁打ちは「質問ラウンド」の繰り返しで進み、**まとめに入るタイミングはユーザーが決める**。

### 1. 観点表を読む

以下の 2 つを読む(後者は存在すれば):

- 汎用観点表:このスキルの `references/checklist.md`
- プロジェクト固有観点表:プロジェクトルートの `.claude/rally-checklist.md`

### 2. 質問ラウンド

- 観点表の 6 観点(入力の異常系/境界/状態と順序/不変条件/解釈の言い直し/スコープ外)を順に検討する
- 各観点について、該当する質問がなければ内部的に「なし」と判断してよい(ユーザーに「なし」の報告は不要)
- 候補の中から**実装を最も左右する最大 5 問**を選ぶ。選別基準は `references/checklist.md` の「重要度判定」を参照
- 二択(または少数の選択肢)で答えられる形に整形する。自由記述を求める質問は避ける
- AskUserQuestion ツールが使える環境ではそれを使い、選択肢を提示する
- 2 ラウンド目以降は、前ラウンドの**保留の解消**(モック・複数案の提示)、回答から新しく
  見えた論点、ユーザーが指定した領域を優先する

### 3. 回答を三分類する

| 分類 | ユーザーの回答例 | 次のアクション |
|---|---|---|
| **確定** | 「B でお願い」「その理解で合ってる」 | 具体例に落とし、E2E(振る舞い仕様)候補として記録 |
| **委任** | 「任せる」「後で見て直す」 | AI が仮決めして明記し、触って確認するフェーズへ回す |
| **保留** | 「どっちがいいか分からない」 | モック・複数案提示で次ラウンドに持ち越す |

「委任」は正式な回答である。委任されたら追い質問せず、仮決め内容を一言で宣言して先へ進む。

### 4. ラウンド末の継続確認

ラウンドの分類が済んだら、現在地を短く示して次をユーザーに聞く:

- 現在地:確定した点/委任(仮決め)/保留/スコープ外を数行で
- 質問:「壁打ちを続けますか、プランにまとめますか?」(AskUserQuestion 可。
  続ける場合に掘りたい領域があれば指定してもらう)

**ユーザーが「まとめて」と言うまでラウンドを繰り返す。** AI の判断で勝手にまとめに
入らない(聞くことが尽きたと思ったら、その旨を伝えた上でユーザーに判断を委ねる)。

### 5. 要件の整合チェック(まとめ指示を受けたら)

1. **矛盾**: ラウンド間で食い違う回答が無いか(後の回答が前の確定と衝突していたら、
   どちらが有効かをユーザーに確認する)。Gherkin シナリオ同士で同じ状況に異なる結果を
   書いていないか
2. **抜け**: 全ラウンドの確定・委任・保留がプランに漏れなく反映されているか。
   保留のまま残っている項目を「保留のまま実装に入ってよいか」明示したか
3. **穴**: 6 観点を最終見直しし、「なし」と判断したまま実は効いてくる観点が無いか
   (確定した仕様同士の組み合わせで新たに生じた未定義がないか)

矛盾・抜け・穴が見つかったら、**まとめを中断して質問ラウンド(手順 2)に戻る**。
見つかった問題を優先論点にしてラウンドを回し(通常どおり最大 5 問・三分類)、
解消したら再びこの整合チェックからやり直す。チェックを通過して初めて次へ進む。
問題がなければ「整合チェック済み(矛盾なし)」と一言添える。

### 6. 技術選定(要件確定後・プラン前)

どんなツール・技術・パッケージで実装するかを整理し、**ユーザーと合意する**
(黙って「お任せ」で決めない):

1. **候補の整理**: 要件を満たす実現手段の候補(2〜3 案)を、採用理由・トレードオフ付きで
   挙げる。依存を増やさない案(標準ライブラリ等)が成立するなら必ず候補に含める
2. **脆弱性チェック**: 外部パッケージを候補に含める場合、選定の時点で候補ごとに調査する:
   - `docs/packages/<パッケージ名>.md` が既に存在すれば調査済みとして読む(再調査しない)
   - **既知脆弱性**: OSV に照会する
     (`curl -s -X POST https://api.osv.dev/v1/query -d '{"package": {"name": "<名前>", "ecosystem": "PyPI"}}'`。
     osv-scanner がインストール済みならそれでもよい)
   - **サプライチェーン**: WebSearch が使えるなら
     `<名前> CVE vulnerability <今年>` と `<名前> security advisory suspicious update <今年>`
     の2系統で検索する(OSV 登録前の乗っ取り更新・不審リリースを拾うため)
   - 調査結果を `docs/packages/<パッケージ名>.md` に保存する(調査日・対象バージョン・
     CVE/脆弱性・不審なアップデート・結論 safe/has_issues)
   - **has_issues の候補は推奨しない**。どうしても必要ならその問題点を明示して
     ユーザーの判断を仰ぐ(黙って採用しない)
   - **OSV の対象は主要 OSS レジストリ(PyPI・npm 等)のみ**。レジストリ外のもの
     (インストーラ/winget で入れる CLI・アプリ、商用 SDK、拡張機能、GitHub 直取得等)は
     OSV に載らないため、WebSearch(CVE・NVD・ベンダーアドバイザリ)だけで調査する
   - 標準ライブラリのみの場合は「外部依存なし・脆弱性チェック対象なし」と明記する
3. **合意**: AskUserQuestion で候補を提示し(脆弱性チェック結果を description に含める)、
   ユーザーの回答で確定する。「任せる」も正式な回答(その場合も選定結果と根拠は宣言する)
4. **技術と要件の整合チェック**: 選定した技術で全ての確定 Gherkin・制約(実行環境・
   配布形態など)が実現できるかを確認する。**実現できない・新たな未定義が生じたら
   質問ラウンド(手順 2)に戻る**

選定結果と理由・捨てた案・脆弱性チェック結果は ADR とインタビュー記録に残す。

### 7. プランの確定

**プランモードが使える環境では、ここでプランモードに入る**(EnterPlanMode)。
まとめはプランとして提示し、ユーザーの承認(プラン承認)を実装開始の指示とみなす。
プランモードが無い環境では従来どおりテキストで提示し、明示の実装指示を待つ。
プランには**技術選定(手順 6)の結果を含める**。

**全ラウンドの結果を統合し**、確定した振る舞いを Given/When/Then で書き出してユーザーに見せる:

```gherkin
Feature: 会員一覧の並び順
  Scenario: 登録日の新しい順に表示される
    Given 田中(3/1 登録)と佐藤(3/5 登録)が存在する
    When 会員一覧を開く
    Then 佐藤が田中より上に表示される
```

これがそのまま E2E 試験コードの元になる。プロジェクトに E2E の置き場所があればそこに追加する。

### 8. インタビュー記録(=プラン資料)の保存

`docs/interviews/YYYY-MM-DD-<短い機能名>.md` に以下を保存する。これが壁打ちの
成果物(プラン)であり、実装フェーズの入力になる。

保存タイミング:
- プランモード利用時:プランモード中はファイルを書けないため、**承認直後・実装より先に**
  保存する(プラン本文と同内容を記録に落とす)
- プランモード非利用時:提示と同時に保存し、それから待機する(未保存のまま待たない)

記録の内容:

- 出した質問(どの観点から出したか)と、選ばなかった観点(「なし」判断の一覧)
- 各回答と三分類の結果
- **要件整合チェックの結果**(問題なし/見つかった問題とその解消。会話だけでなく記録に残す)
- **確定した Gherkin の全文**(「〜を参照」と書かず、シナリオ本文をこの記録に埋め込む。
  記録単体で仕様が読めることが目的)
- **技術選定**(採用した技術・パッケージ、理由、捨てた案、脆弱性チェック結果)
- 委任された点と AI の仮決め内容
- 保留された点

この記録は消耗品(維持義務なし)だが、ADR の材料になり、後日「観点漏れだったか」を
振り返る一次資料になる。

### 9. 承認待ちと実装開始

- **プランモード利用時**:プラン承認が実装開始の指示。承認されたら手順 8 の記録保存を
  行ってから実装に入る。**修正要求・非承認なら壁打ち(手順 2)に戻る**
- **プランモード非利用時**:手順 8 の保存を済ませた上で、要約(確定 Gherkin/技術選定/
  委任/保留/スコープ外)を提示して待機する。実装を開始するのは「実装して」等の明示指示があったときだけ

どちらの場合も、AI の判断で実装に入ることはない。

## 実装フェーズ:標準試験バッテリーの自動展開

実装するときは、確定した Gherkin の E2E 化に加えて、`references/test-battery.md` の
該当カテゴリ(境界値・入力異常系・データ頑健性・冪等性・不変条件・出力契約など)を
**ユーザーに確認せず自動で** E2E/単体に展開する。機械的に列挙できる試験を人間の
プローブ待ちにしない。

## 実装後:人間が触るフェーズへの接続

実装が終わったら、人間が実際に触って確認するフェーズに入る。ユーザーがリモート
(モバイルアプリ等)から参加していて直接操作できない場合は、**AI が代わりに実際の
コマンド・操作を実行し、加工していない実行出力をそのまま見せる**。ユーザーはその出力を
見て反例を出す(実行は AI、判定は人間、という分担)。人間に見せるのは仕様の解釈・
使い心地の確認であり、バッテリーで潰せる機械的な欠陥を人間に発見させたら負けである。

バグ・期待と違う挙動が報告されたら、修正の前に promote-bug スキルで再現試験を
E2E へ昇格させる。

### 実装・トライアンドエラー中に未定義の仕様に遭遇したら

実装漏れ・エラー・想定外の状況で「どう振る舞うべきか」が既存の記録・Gherkin から
導けないときは、**AI の推測で埋めずミニインタビューに戻る**:

- 取りうる扱いを選択肢(1〜2問)で提示し、通常どおり三分類で回答を受ける
- 確定したらインタビュー記録(プラン資料)と ADR に追記し、E2E に固定してから実装を続ける
- ただし**外から見えない内部実装の都合は委任枠で仮決めしてよい**。質問に戻るのは
  観測可能な振る舞いが未定義のときだけ(質問攻めにしない)

これは**単体試験・バッテリー展開中も同じ**。テストの期待値が仕様から導けないのは
「仕様が未定義」のシグナルである。観測可能な振る舞いのアサーションを推測で書くと、
推測がテストとして固定化される(現状追認の亜種)。書けない期待値に出会ったら、
観測可能なもの→ミニインタビュー、内部都合→委任枠、で処理する。

## 観点表を育てる

インタビュー後に以下が起きたら、観点表への追記を提案する:

- 質問が的外れだった(ユーザーに「そこじゃない」と言われた)
- 観点漏れがあった(聞かなかったことが後でバグ・手戻りになった)

追記先の使い分け:

- どのプロジェクトでも通用する教訓 → プラグインの `references/checklist.md` に
  **一般形へ蒸留して**項目だけ追記する(プロジェクト名・日付・経緯は書かない。
  この表は配布物であり、インタビュー中に読まれるため過去案件の事情が質問に漏れる)
- このプロジェクト特有の事情・経緯 → `.claude/rally-checklist.md` の育成ログ
  (なければ作成する。形式: 日付・何が起きたか・教訓・反映先)
