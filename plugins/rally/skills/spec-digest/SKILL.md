# spec-digest:人間可読の仕様書を生成する

rally では「最新の正しい仕様」は E2E 試験コードだが、初見の人には読みにくい。
このスキルは E2E と ADR から人間可読の仕様書 `docs/spec.md` を導出する。

## 原則

1. **フォーマットは固定。** `references/format.md` の見出し番号・名称・順序を絶対に変えない。
   再生成のたびに構成が変わると読者が混乱する。機能が増えたらセクション 3 のサブ節が
   増えるだけ、が正しい変化。
2. **書いてよいのは E2E で検証されている振る舞いだけ。** コードやインタビュー記録から
   「たぶんこう動く」を書かない。E2E に無い振る舞いに言及する場合は
   「7. 未確定・保留事項」に置く。
3. **spec.md は派生物。** 手で編集しない・維持しない。仕様が変わったら再生成する。
   古い spec.md と E2E が食い違ったら、正は常に E2E。

## 手順

1. `tests/e2e/`(および `.feature` があればそれ)を読み、検証されている振る舞いを
   機能別に整理する
2. `docs/adr.md` を読み、主な設計判断を要約する(全件でなく、利用者の理解に効くものを選ぶ)
3. `docs/interviews/` の最新記録から「スコープ外」「保留」を拾う
4. `references/format.md` のテンプレートに流し込み、`docs/spec.md` に上書き保存する
5. 生成後、E2E との突き合わせを一度行う(spec.md に書いたのに対応する試験が無い記述、
   逆に主要機能なのに spec.md に無い試験、があれば修正する)

## 品質基準

- **可読性**: アプリを知らない人が spec.md だけで「何ができるか・どう使うか・
  何をするとどうなるか」を把握できる
- **整合性**: 全ての記述が E2E の試験に対応している(生成元セクションの参照が正しい)
